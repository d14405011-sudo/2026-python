## 問題回答

### 1. 哪個操作最耗時？你認為原因是什麼？

根據本次測量結果，`read_json`（0.002562s）與 `read_csv`（0.002442s）的耗時非常接近，幾乎可被視為最耗時的操作，但彼此差距極小（僅幾十微秒）。主要原因是目前的測試資料集非常小（僅 437 筆資料，篩選後 189 筆）。在資料規模較小的情境下，I/O 操作和建立連線等「開銷（overhead）」佔據了大部分時間，而各種格式本身在解析速度上的差異反而難以彰顯。若將資料量等比放大，通常 `write_xml` 會因為需要逐一產生節點物件、構建並在記憶體中維護龐大的 DOM 樹（`ET.SubElement` 等操作）而成為最耗時的操作。相比之下，`read_csv` 的瓶頸在於它需要用迴圈逐行讀取、分割字串，並在每次讀取為字典時動態分配記憶體。

### 2. read_csv 比 read_json 快還是慢？與課堂 U01 的比較實驗結果一致嗎？

在本次結果中：`read_csv`（0.002442s）略**快於** `read_json`（0.002562s），然而差異極小（大約 0.00012s）。

這與課堂 U01 所觀察到「JSON 讀取通常最快、CSV 居次」的一般預期有些微出入。在課堂的比較實驗中（1000 筆隨機資料），透過原生的 C 語言擴充模組，`json.loads()` 能夠極有效率地反序列化資料。本次 `read_csv` 看似略快，可能由於兩者讀取的是檔案而不是純記憶體字串，並且本實驗資料量太小（僅幾百筆），使得隨機的系統 I/O 開銷、OS 快取機制或甚至 JIT（即時編譯）冷啟動狀態，掩蓋了純粹的 CPU 解析時間差異。可以視為在這個資料量下，兩者讀取時間非常接近。

### 3. write_xml 比 write_json 快還是慢？原因為何？

本次結果顯示：`write_xml`（0.001698s）明顯**快於** `write_json`（0.002385s）。

通常我們預期輸出 XML 會比 JSON 慢，但這次反而是 `write_xml` 較快。推測這有幾個原因：第一，為了產生人類易讀的 JSON 檔案，我們在調用 `json.dump` 時使用了 `indent=2` 和 `ensure_ascii=False`，這會迫使 Python 對所有字串執行額外的排版（計算空白縮排及換行）與字元編碼檢查，這拖慢了 JSON 的寫入速度。第二，這批資料的結構相當單純，每個 `student` 只有少數四個欄位作為節點屬性，沒有複雜的層狀子結構。此種簡單的格式搭配 C 語言寫成的 `xml.etree.ElementTree` 模組，使得 XML 快速建立 DOM 樹並輸出成字串的速度優勢超越了加入了格式化負擔的 JSON。

### 4. 如果資料筆數從 100 增加到 10000，你預期各函式耗時如何變化？

若資料筆數放大了至少 100 倍（從約百筆增加到 10000 筆以上），所有函式的耗時皆呈現大致線性的 $O(n)$ 成長，但它們各自增長斜率大不相同：
- **`write_xml`**：預期耗時會急劇增加。因為它需要針對每筆紀錄重複呼叫 `ET.SubElement`，產生許多小型的 Python 記憶體物件，並且樹狀結構完全保存在記憶體中，當元素逼近上萬個時，記憶體分配與建置樹狀結構的開銷會非常大。
- **`read_csv` vs `read_json`**：`read_json` 將會明顯贏過 `read_csv`。一萬筆的 JSON 字串可以交由優化過的底層 C 函式一次性批量解析，而 `read_csv` 仍須不斷地針對每一行迭代、分割並建構新的字典。
- **`write_json`**：耗時的增加幅度會居中。比起 `write_xml` 需要耗費大量的記憶體處理物件節點，將 Dict 直接序列化為 JSON 的負擔相對較輕，即使有 `indent`，整體的擴展性仍然優於產生 XML。
